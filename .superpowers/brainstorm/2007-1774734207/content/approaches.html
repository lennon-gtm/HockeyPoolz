Three ways to build HockeyPoolz

Pick the foundation — this decision shapes everything else

A

Next.js + Supabase Recommended

One platform, everything included. Supabase handles auth (Google + Apple built-in), Postgres database, and real-time score updates. Next.js covers frontend and API in one codebase. Deploy to Vercel in minutes.

Pros

  • Fastest path to Alpha
  • Google/Apple SSO out of the box
  • Real-time score sync built in
  • Scales to hundreds of leagues no problem
  • Stripe payment layer plugs in later easily

Cons

  • Some Supabase lock-in (mitigated — it's open-source Postgres)
  • Advanced real-time at massive scale needs tuning
B

React SPA + Node.js API + PostgreSQL

Traditional, fully custom stack. Separate React frontend and Node.js/Express backend. Auth via Auth0. Real-time via Socket.io. Full control over every layer.

Pros

  • No platform lock-in
  • Maximum flexibility
  • Battle-tested patterns

Cons

  • More to build and maintain
  • Auth0 adds cost and complexity early
  • Real-time infra requires more setup
  • 2–3x slower to reach Alpha
C

Next.js + Dedicated Stats Microservice

Split the stats concern out early. Next.js app handles all user-facing features. A separate Node.js service polls the NHL API, caches in Redis, and pushes score updates. Most scalable architecture.

Pros

  • Stats service swappable independently (NHL API → SportsRadar)
  • Best separation of concerns
  • Easiest to scale the stats layer separately

Cons

  • Two services to deploy and maintain from day one
  • Premature for Alpha scale
  • More DevOps overhead
